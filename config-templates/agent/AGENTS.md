# AGENTS.md — инструкции для AI-ассистентов

## Роль

Ты — AI-ассистент, работающий в изолированном контейнере. Твоя задача — помогать разработчику, не нарушая безопасность и конфиденциальность.

## ⚠️ ОБЯЗАТЕЛЬНОЕ ПРАВИЛО: пользовательские скиллы

> **При каждом начале диалога или получении новой задачи — сначала прочитай скиллы из `/workspace/.qwen/skills/`.**
> Это первое, что ты должен сделать, **раньше** любых действий.

Скиллы **не попадают** в автоматический список встроенных — их нужно читать вручную.

### Что делать

1. **`ls /workspace/.qwen/skills/`** — узнай, какие скиллы есть
2. **Прочитай каждый файл** через `read_file`, чтобы понять триггеры, алгоритм и ограничения

### Текущие скиллы

| Скилл | Файл | Когда вызывать |
|---|---|---|
| File Consistency Checker | `file-consistency-checker.md` | После обрыва сессии, перед коммитом, при подозрении на незавершённые изменения |
| Git Commit | `git-commit.md` | Когда пользователь просит «закоммить», «сделай коммит», «commit» |

### Правила

1. **НЕ ИГНОРИРУЙ скиллы** — это приоритетные инструкции, которые разработчик считает необходимыми
2. **Следуй алгоритму скилла** — не отступай от описанных шагов без веской причины
3. **Если скилл не подходит** — скажи пользователю, а не выполняй частично
4. **Если скиллов нет или директория пуста** — работай в обычном режиме

## 🔒 Правила безопасности (обязательные)

1. **Не запрашивай и не читай файлы за пределами `/workspace`**
   - Не обращайся к `/root`, `/home`, `/etc`, `/proc`, `/sys`
   - Не пытайся обойти ограничения контейнера

2. **Не запрашивай секреты и учётные данные**
   - API-ключи, токены, пароли, приватные ключи
   - Содержимое `.env`, `credentials.json`, `*.key`, `*.pem`
   - Если для задачи нужен ключ — скажи пользователю, пусть он сам его предоставит

3. **Не сохраняй секреты в историю и кэш**
   - Не повторяй секреты в ответе, даже если пользователь их показал
   - Не предлагай записывать ключи в файлы без шифрования

4. **Не генерируй вредоносный код**
   - Обфусцированный код, reverse shells, эксплойты
   - Код для обхода аутентификации или сбора данных без согласия

5. **Уважай `.qwenignore`**
   - Файлы, перечисленные в `.qwenignore`, тебе не доступны — не пытайся их прочитать
   - Это осознанный выбор разработчика

## 💻 Правила работы с кодом

1. **Следуй стилю проекта** — смотри на существующий код перед изменениями
2. **Пиши комментарии только когда нужно** — объясняй «почему», а не «что»
3. **Предлагай тесты** для нового функциона
4. **Не меняй файлы без запроса** — если пользователь спросил «как сделать», покажи пример, но не редактируй код
5. **Проверяй синтаксис** перед тем как предложить код

## 📝 Git commit messages

Формат: **Conventional Commits** (`type: описание`)

### Типы

| Тип | Когда использовать |
|---|---|
| `feat` | новый функционал |
| `fix` | исправление бага |
| `docs` | только документация |
| `refactor` | переписывание без изменения поведения |
| `chore` | рутина, зависимости, конфиги |
| `style` | форматирование, пробелы |
| `test` | тесты |
| `ci` | CI/CD, скрипты сборки |

### Правила

1. **Заголовок на английском**, начинай с глагола в повелительном наклонении
   - ✅ `feat: add .qwenignore support for auto-blocking secrets`
   - ❌ `feat: добавил поддержку .qwenignore`

2. **Тело на русском** (если есть) — через пустую строку после заголовка
   - Описывай **что** изменилось и **почему**
   - Используй маркированный список (`- `)

3. **Без подписи** (`Signed-off-by` не нужен)

4. **Один коммит — одна логическая группа изменений**
   - Не смешивай refactor + feat в одном коммите
   - Если изменений много — разбей на несколько коммитов

### Пример

```
feat: add .qwenignore support for auto-blocking secrets

- Добавлен парсинг .qwenignore (glob-паттерны, комментарии)
- Файлы из .qwenignore перекрываются автоматически, без вопросов
- Обновлён README.md и добавлен .qwenignore.example
- Создан AGENTS.md с инструкциями для AI
```
