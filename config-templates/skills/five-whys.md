# Skill: Five Whys (Пять почему)

Систематический метод поиска первопричин проблем. Помогает не останавливаться на симптомах, а докопаться до корня.

## Когда использовать

- При поиске причин бага или неожиданного поведения
- Когда нужно разобраться в проблеме глубже поверхностного «починил»
- По запросу пользователя: «почему это не работает?», «разберись», «найди причину»
- Когда предыдущее исправление не помогло — проблема вернулась
- Перед тем как предлагать решение — сначала понять причину

## Алгоритм

### 1. Сформулируй проблему

Запиши проблему **конкретно** и **фактологически**:

```
Проблема: {что произошло, где, когда}
Ожидание: {что должно было произойти}
Контекст: {окружение, версия, последние изменения}
```

### 2. Задай «Почему?» пять раз

Для каждого ответа задавай следующий «Почему?». Иди от симптома к системной причине.

```
Почему #1: {почему возникла проблема?}
  → Ответ: {первая причина}

Почему #2: {почему возникла эта причина?}
  → Ответ: {вторая причина}

Почему #3: {почему это произошло?}
  → Ответ: {третья причина}

Почему #4: {почему система допустила это?}
  → Ответ: {четвёртая причина}

Почему #5: {почему процесс/архитектура не защитили?}
  → Ответ: {корневая причина}
```

### 3. Найди контрмеру

Для каждой корневой причины предложи **системное** решение:

| Уровень | Тип решения | Пример |
|---|---|---|
| 🟢 Быстрое | Исправить симптом | «Перезапустить сервис» |
| 🟡 Тактическое | Устранить причину | «Починить конфиг» |
| 🔴 Системное | Защитить от повторения | «Добавить тест/валидацию» |

### 4. Задокументируй

Сохрани результат в `/workspace/.qwen/five-whys/`:

```
/workspace/.qwen/five-whys/YYYY-MM-DD-краткое-описание.md
```

Формат отчёта:

```markdown
# Five Whys: {краткое описание проблемы}

**Дата:** YYYY-MM-DD
**Автор:** {кто провёл анализ}

## Проблема
{что случилось}

## Ожидание
{что должно было быть}

## Контекст
{версия, окружение, последние изменения}

## Анализ

| Почему # | Вопрос | Ответ |
|---|---|---|
| 1 | Почему ...? | ... |
| 2 | Почему ...? | ... |
| 3 | Почему ...? | ... |
| 4 | Почему ...? | ... |
| 5 | Почему ...? | ... |

## Корневая причина
{одним предложением}

## Контрмеры

| Уровень | Действие | Статус |
|---|---|---|
| 🟢 Быстрое | ... | [ ] |
| 🟡 Тактическое | ... | [ ] |
| 🔴 Системное | ... | [ ] |
```

## Правила

1. **Не останавливайся на первом ответе** — «скрипт упал» ≠ причина, это симптом
2. **Ищи системные причины** — «забыл» ≠ причина, «нет проверки» ≠ причина, а «нет процесса валидации» — уже ближе
3. **Не обвиняй людей** — фокусируйся на процессах, инструментах, архитектуре
4. **Если дошёл до 5 — спроси, достаточно ли** — иногда хватает 3, иногда нужно 7
5. **Если ветвится** — проблема имеет несколько корневых причин, исследуй каждую
6. **Если зашёл в тупик** — собери больше данных (логи, diff, статус) и начни заново

## Пример

```
Проблема: qcc не видит скиллы в .qwen/skills/

Почему #1: Почему qcc не видит скиллы?
  → glob не нашёл файлы в .qwen/skills/

Почему #2: Почему glob не нашёл?
  → директория .qwen/ добавлена в .gitignore

Почему #3: Почему .qwen/ в .gitignore?
  → чтобы не коммитить локальные AI-конфиги и сессии

Почему #4: Почему это ломает чтение скиллов?
  → glob-утилита уважает .gitignore и пропускает .qwen/

Почему #5: Почему нет fallback на прямой путь?
  → скиллы читаются только через glob, без read_file по известному пути

Корневая причина: единый механизм чтения скиллов через glob, который не учитывает .gitignore

Контрмеры:
  🟢 Прочитать скиллы напрямую: read_file .qwen/skills/*.md
  🟡 Добавить в glob параметр respect_git_ignore: false для .qwen/skills/
  🔴 Перенести скиллы из .qwen/skills/ в config-templates/skills/ (уже там) и читать оттуда
```

## Не делай

- ❌ Не останавливайся на «человек забыл» — ищи почему не было напоминания/проверки
- ❌ Не предлагай решения до анализа причины
- ❌ Не смешивай несколько проблем в одном анализе
- ❌ Не выдумывай ответы — если не знаешь, собери больше данных
- ❌ Не удаляй отчёты — они полезны для истории и обучения
