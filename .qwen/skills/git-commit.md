# Skill: Git Commit

## Trigger

Применяй этот скилл когда пользователь просит:
- «закоммить», «сделай коммит», «commit»
- «подготовь коммит», «prepare a commit»
- «закоммить изменения», «commit changes»
- «что в коммите», «what's in the commit»

## Rules

### 1. Сначала собери информацию

Всегда выполняй **одной командой**:
```
git status && git diff HEAD && git log -n 3 --oneline
```

- `git status` — какие файлы изменены, что staged
- `git diff HEAD` — все изменения (включая unstaged)
- `git log -n 3 --oneline` — стиль последних коммитов

### 2. Проанализируй изменения

- Определи **тип** по Conventional Commits: `feat`, `fix`, `docs`, `refactor`, `chore`, `style`, `test`, `ci`
- Если изменения затрагают несколько несвязанных тем — предложи разбить на несколько коммитов
- Если есть unstaged изменения, которые **не должны** попасть в коммит — предупреди
- **Проверь diff на иероглифы (CJK-символы).** Проект не должен содержать иероглифов — если найдены, сообщи пользователю и предложи исправить до коммита

### 3. Стажь файлы

```
git add <файлы>
```

- Ставь только файлы, относящиеся к этому коммиту
- Не используй `git add -A` без подтверждения пользователя
- Если пользователь не уточнил — покажи что будет staged и спроси

### 4. Предложи сообщение коммита

**Всегда предлагай черновик.** Не спрашивай «какое сообщение?» — дай готовый вариант.

Формат (из AGENTS.md):
- **Заголовок**: английский, `type: описание`, глагол в повелительном наклонении
- **Тело**: русское, через пустую строку, маркированный список
- Каждая строка списка: **что сделано** + кратко **почему** (мотивация в нескольких словах)
- **Без подписи**

Пример:
```
feat: add .qwenignore support for auto-blocking secrets

- Добавлен парсинг .qwenignore, чтобы исключить секреты из контекста AI
- Файлы из .qwenignore перекрываются автоматически для защиты учётных данных
- Обновлён README.md, так как добавлена новая функциональность
```

### 5. Спроси подтверждение

Покажи пользователю:
- Что будет в коммите (файлы)
- Сообщение коммита (черновик)
- Спроси: «Коммитим?» или «Поправить что-то?»

### 6. Выполнить коммит

```
git commit -m "заголовок

тело"
```

### 7. Проверь результат

```
git status && git log -n 1 --oneline
```

Убедись что коммит прошёл и working tree clean.

## Edge cases

| Ситуация | Действие |
|---|---|
| Нечего коммитить | Скажи «working tree clean, нечего коммитить» |
| Нет staged файлов | Предложи `git add` или спроси какие файлы |
| Конфликт merge/rebase | Не коммить, скажи пользователю разрешить конфликт |
| Нет git identity | Предложи `git config user.name/email` |
| Изменения в .env или секретах | **Предупреди** что секреты могут попасть в историю |
| Повышение версии (VERSION, package.json и т.п.) | После коммита создай **тег** с новой версией: `git tag v<версия>` и сообщи пользователю |

## Повышение версии

Если в коммите изменяется файл версии (`VERSION`, `package.json/version` и аналогичные):

1. Определи новую версию из diff
2. После успешного коммита создай тег: `git tag v<новая_версия>`
3. В сообщении коммита укажи версию: `chore: bump version to v<версия>`
4. Сообщи пользователю: «Создан тег v<версия>». Для отправки в remote нужно выполнить `git push origin v<версия>`

## Не делай

- ❌ Не коммить без подтверждения пользователя
- ❌ Не использовать `git commit --amend` без явного запроса
- ❌ Не пушить в remote без явного запроса
- ❌ Не игнорировать unstaged изменения
- ❌ Не коммить секреты (предупреди пользователя)
