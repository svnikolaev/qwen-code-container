## ЧАСТЬ I. ТЕХНИЧЕСКОЕ ЗАДАНИЕ НА РАЗРАБОТКУ

---

### 1. Общие сведения

#### 1.1. Назначение проекта

Разработать оболочку (launcher) для безопасного запуска AI-ассистента **Qwen Code** в изолированном Docker-контейнере. Основная задача — предотвратить утечку секретов (API-ключей, токенов, приватных ключей) из рабочей директории разработчика в AI-ассистент, а также обеспечить чистоту проекта от служебных файлов конфигурации.

#### 1.2. Цели

- **Безопасность:** автоматическое обнаружение и перекрытие файлов с секретами перед монтированием в контейнер
- **Изоляция:** AI-ассистент работает только с файлами проекта, без доступа к домашней директории и системным файлам хоста
- **Чистота проекта:** все служебные данные (кэш npm, конфиги, скиллы) хранятся в глобальной директории, а не в рабочей папке
- **Удобство:** единая команда для запуска из любого проекта

#### 1.3. Область применения

Инструмент предназначен для разработчиков, использующих AI-ассистента Qwen Code в Docker-контейнере и желающих защитить свои секреты от попадания в контекст AI.

### 2. Требования к системе

#### 2.1. Функциональные требования

##### 2.1.1. Обнаружение секретов

Система должна автоматически сканировать рабочую директорию проекта и выявлять файлы, потенциально содержащие секреты.

**Обязательные категории для поиска:**

| Категория            | Паттерны                                       |
| -------------------- | ---------------------------------------------- |
| Файлы окружения      | `.env`, `.env.local`, `.env.*.local`, `.env.*` |
| Ключи и сертификаты  | `*.key`, `*.pem`, `*.crt`, `*.pk`              |
| Токены и пароли      | `*.token`, `*.password`                        |
| Учётные данные       | `credentials.json`, `client_secret*.json`      |
| SSH-ключи            | `id_rsa`, `id_dsa`                             |
| Секретные директории | `.secret`, `secrets/`, `secret/`               |

**Требования:**

- Глубина сканирования — не менее 2 уровней вложенности от корня проекта
- Файлы-шаблоны (например, `.env.example`) не должны считаться секретами
- Список паттернов должен быть расширяемым через переменные окружения
- Результаты сканирования дедуплицируются и сортируются

##### 2.1.2. Автоматическое перекрытие (.qwenignore)

Система должна поддерживать файл `.qwenignore` в корне проекта с glob-паттернами для автоматического перекрытия файлов.

**Требования:**

- Формат: один паттерн на строку
- Поддержка комментариев (строки, начинающиеся с `#`)
- Поддержка glob-масок (`*.key`, `.env.*`)
- Поддержка директорий (`secrets/` — все файлы внутри)
- Поддержка относительных путей (`config/secrets.yml`)
- Файлы, совпавшие с паттернами, перекрываются автоматически, без вопросов пользователю

##### 2.1.3. Интерактивный выбор перекрытия

Для секретов, не покрытых `.qwenignore`, система должна запросить решение у пользователя.

**Требования:**

- Для каждого найденного секрета задать вопрос: «Перекрыть при монтировании?»
- Ответ по умолчанию — «да» (Y)
- Выбор пользователя сохраняется и используется при повторных запусках
- Ключ сохранения — абсолютный путь к проекту
- Формат хранения — JSON-файл с маппингом «путь проекта → список перекрытых файлов»

##### 2.1.4. Режимы запуска

| Режим        | Описание                                            |
| ------------ | --------------------------------------------------- |
| Стандартный  | Запуск с защитой, использование сохранённого выбора |
| Refresh      | Переспросить о перекрытии всех секретов заново      |
| Show-blocked | Показать текущие перекрытые файлы и выйти           |
| Model        | Показать текущую модель и выйти                     |
| Debug        | Показать полную команду запуска и выйти             |
| Help         | Вывести справку                                     |

##### 2.1.5. Управление моделью

- Возможность задать версию AI-модели через команду
- Модель сохраняется между запусками
- Модель передаётся как аргумент при каждом запуске контейнера

##### 2.1.6. Установка и инициализация

Система должна предоставлять команды для:

- Установки глобальной команды запуска (symlink в `~/.local/bin`)
- Инициализации глобальных директорий конфигурации
- Копирования шаблонов скиллов и конфигов
- Настройки OAuth-авторизации для проекта
- Проверки и установки зависимостей (`jq`)
- Очистки глобальной конфигурации
- Удаления глобальной команды

##### 2.1.7. Git-интеграция

- Git-конфигурация (имя, email) с хоста должна автоматически передаваться в контейнер
- Файл `AGENTS.md` из корня проекта должен монтироваться в контейнер как системный промпт для AI
- Контейнер должен использовать те же имя и email для git-коммитов, что и хост

##### 2.1.8. Глобальные скиллы

- Система должна поддерживать глобальные скиллы (наборы инструкций для AI), общие для всех проектов
- Скиллы из шаблонов проекта должны копироваться в глобальную директорию при установке
- Скиллы монтируются в контейнер в режиме read-only

#### 2.2. Нефункциональные требования

##### 2.2.1. Безопасность контейнера

- Запуск без привилегий (`no-new-privileges`)
- Запуск от текущего пользователя хоста (не root)
- SELinux/AppArmor метки должны быть корректно обработаны
- Временные директории внутри контейнера не должны записываться на хост

##### 2.2.2. Совместимость

- Поддержка Docker и Podman
- Автоматическое определение runtime
- Корректная работа на Linux (Ubuntu, Fedora, Arch) и macOS

##### 2.2.3. Чистота проекта

В рабочей директории проекта **не должны** создаваться:

- `.npm/`
- `.config/`
- `.qwen/`

Все служебные данные выносятся в глобальную директорию.

##### 2.2.4. Миграция

При обновлении системы должна поддерживаться миграция старой конфигурации в новую директорию (если старая директория существует, а новая — нет).

### 3. Архитектура

#### 3.1. Компоненты системы

| Компонент                     | Назначение                                                                    |
| ----------------------------- | ----------------------------------------------------------------------------- |
| Launcher-скрипт               | Главный скрипт: сканирует секреты, управляет перекрытием, запускает контейнер |
| Entry-point скрипт            | Скрипт внутри контейнера: настраивает окружение перед запуском AI             |
| Makefile                      | Набор команд для управления жизненным циклом                                  |
| Шаблоны конфигурации          | Скиллы и конфиги, копируемые при установке                                    |
| Файл перекрытия (.qwenignore) | Пользовательский файл с паттернами автоматического перекрытия                 |
| Системный промпт (AGENTS.md)  | Инструкции для AI-ассистента                                                  |

#### 3.2. Глобальная конфигурация

Все общие данные хранятся в единой глобальной директории (XDG Base Directory: `~/.config/<имя-проекта>/`):

| Поддиректория / файл     | Назначение                                        |
| ------------------------ | ------------------------------------------------- |
| `npm/`                   | Кэш npm (шареный между проектами)                 |
| `config/`                | Общие конфиги (шареные)                           |
| `skills/`                | Глобальные скиллы для AI-агентов                  |
| `projects/<hash>/.qwen/` | Изолированные конфиги Qwen для каждого проекта    |
| `blocked-paths.json`     | Сохранённый выбор перекрытия секретов по проектам |
| `model`                  | Выбранная версия модели                           |
| `cmd`                    | Кэш имени команды внутри контейнера               |

Хэш проекта вычисляется по абсолютному пути рабочей директории.

#### 3.3. Структура проекта

```text
<проект>/
├── bin/
│   ├── launcher-скрипт       # Главный скрипт запуска
│   └── entrypoint-скрипт     # Entry-point контейнера
├── config-templates/
│   ├── qwen/                 # Шаблоны конфигов Qwen
│   └── skills/               # Глобальные скиллы
├── AGENTS.md                 # Инструкции для AI
├── Makefile                  # Команды управления
├── .env.example              # Пример переменных окружения
└── .qwenignore.example       # Пример файла перекрытия
```

#### 3.4. Схема монтирования в контейнер

| Хост                                 | Контейнер                             | Описание                     |
| ------------------------------------ | ------------------------------------- | ---------------------------- |
| `<проект>`                           | `/workspace`                          | Рабочая директория           |
| `<глобальный>/npm`                   | `/root/.npm`                          | Кэш npm                      |
| `<глобальный>/config`                | `/root/.config`                       | Общие конфиги                |
| `<глобальный>/projects/<hash>/.qwen` | `/root/.qwen`                         | Конфиг Qwen                  |
| `<глобальный>/projects/<hash>/.qwen` | `/workspace/.qwen`                    | Конфиг Qwen в проекте        |
| `<глобальный>/skills`                | `/root/.qwen/shared-skills:ro`        | Скиллы (read-only)           |
| `AGENTS.md`                          | `/root/AGENTS.md:ro`                  | Системный промпт (read-only) |
| `entrypoint-скрипт`                  | `/usr/local/bin/git-entrypoint.sh:ro` | Entry-point (read-only)      |
| Перекрытые файлы                     | `-v /workspace/<файл>`                | Пустой volume поверх секрета |

### 6. Сценарии использования

#### 6.1. Первичная установка

1. Клонировать репозиторий
2. Выполнить `make install` — создаются глобальные конфиги, symlink команды
3. Выполнить `make setup` — настраивается OAuth
4. Убедиться, что `~/.local/bin` в `PATH`

#### 6.2. Запуск в проекте

1. Перейти в директорию проекта
2. Выполнить `qwen`
3. При первом запуске — ответить на вопросы о перекрытии секретов
4. Контейнер запущен, AI-ассистент готов к работе

#### 6.3. Настройка .qwenignore

1. Скопировать `.qwenignore.example` в `.qwenignore`
2. Отредактировать паттерны под свой проект
3. При следующем запуске файлы по паттернам перекроются автоматически

### 7. Формат сообщений Git commit

Проект использует **Conventional Commits**:

```text
type: описание на английском

- Тело на русском (маркированный список)
- Описывает что и почему изменилось
```

| Тип        | Описание                              |
| ---------- | ------------------------------------- |
| `feat`     | Новый функционал                      |
| `fix`      | Исправление бага                      |
| `docs`     | Только документация                   |
| `refactor` | Переписывание без изменения поведения |
| `chore`    | Рутина, зависимости, конфиги          |
| `style`    | Форматирование, пробелы               |
| `test`     | Тесты                                 |
| `ci`       | CI/CD, скрипты сборки                 |

**Правила:**

- Заголовок на английском, начинается с глагола в повелительном наклонении
- Тело на русском, через пустую строку
- Без подписи (`Signed-off-by`)
- Один коммит — одна логическая группа изменений

#### 10.2. Реализация обнаружения секретов

**Файл:** `bin/qwen-run`, функция `find_secrets()`

**Механизм:**

```bash
# Встроенные паттерны (массив DEFAULT_PATTERNS)
DEFAULT_PATTERNS=(
    ".env" ".env.local" ".env.*.local" ".env.*"
    "*.key" "*.pem" "*.crt" "*.pk"
    "secrets" "secret" "credentials.json" "client_secret*.json"
    ".secret" "id_rsa" "id_dsa" "*.token" "*.password"
)

# Исключения (массив SECRET_EXCLUSIONS)
SECRET_EXCLUSIONS=(".env.example")
```

- Сканирование через `find` с `-maxdepth 2`
- Результаты проходят через `sort -u` для дедупликации
- Файлы из `SECRET_EXCLUSIONS` исключаются после поиска
- Пользователь может переопределить паттерны через `QWEN_SECRET_PATTERNS`

#### 10.3. Реализация .qwenignore

**Файл:** `bin/qwen-run`, функции `parse_qwenignore()`, `matches_qwenignore()`, `find_qwenignore_files()`

**Алгоритм:**

1. `parse_qwenignore()` — читает файл, удаляет комментарии (`#`) и пустые строки
2. `find_qwenignore_files()` — ищет файлы, совпадающие с паттернами (поддержка директорий `*/` и glob `*`)
3. `matches_qwenignore()` — проверяет конкретный путь против паттернов:
   - Сравнение по имени файла (`$filename == $pattern`)
   - Сравнение по полному пути (`$filepath == $pattern`)
   - Поддержка путей с `/` (`config/secrets.yml`)
   - Поддержка директорий (`secrets/` → `$dir_name/*`)

Найденные файлы добавляются в `qwenignore_auto_blocked` и исключаются из интерактивного опроса.

#### 10.4. Реализация интерактивного выбора

**Файл:** `bin/qwen-run`, функции `ask_block()`, `save_blocked_for_project()`, `load_blocked_for_project()`, `project_already_asked()`

**Хранение:** `~/.config/qwen-code-docker/blocked-paths.json`

**Формат JSON:**

```json
{
  "/home/user/my-project": [".env", "*.key", "secrets/db.json"],
  "/home/user/another-project": []
}
```

**Логика:**

1. При первом запуске: для каждого секрета (не в `.qwenignore`) — вопрос `[Y/n]`
2. Функция `project_already_asked()` проверяет, есть ли ключ проекта в JSON (даже если массив пустой)
3. При повторном запуске: загружается сохранённый список, вопросы не задаются
4. `--refresh` сбрасывает логику — вопросы задаются заново
5. `QWEN_AUTO_BLOCK_SECRETS=1` — блокирует все секреты без вопросов
6. `QWEN_REFRESH_BLOCK=1` — аналог `--refresh` через переменную окружения

#### 10.5. Реализация перекрытия файлов

**Файл:** `bin/qwen-run`, функция `build_overlay_opts()`

**Механизм:** каждый перекрытый файл монтируется как пустой volume:

```bash
-v /workspace/.env -v /workspace/secrets/db.json -v /workspace/*.key
```

Docker монтирует пустой anonymous volume поверх существующего файла, делая его недоступным внутри контейнера.

#### 10.6. Реализация запуска контейнера

**Файл:** `bin/qwen-run`, функция `main()`

**Определение runtime:**

```bash
detect_runtime() {
    # Проверяет docker --version на наличие "podman"
    # Возвращает "docker", "podman" или "unknown"
}
```

**Опции пользователя:**

| Runtime | Опции                                              |
| ------- | -------------------------------------------------- |
| Docker  | `--user $(id -u):$(id -g) --group-add keep-groups` |
| Podman  | `--userns=keep-id --group-add keep-groups`         |

**Определение команды:** `detect_qwen_cmd()` пробует `which qwen`, затем `which qwen-code` внутри образа. Результат кэшируется в `~/.config/qwen-code-docker/cmd`.

**Полная команда docker:**

```bash
docker run --rm -it \
    $USER_OPTS \
    --security-opt label=disable \
    --security-opt no-new-privileges:true \
    -v "$PROJECT_DIR":/workspace \
    ${overlay_opts} \
    $GLOBAL_MOUNTS \
    $SKILLS_MOUNT \
    $GIT_MOUNT \
    $TMPFS_MOUNTS \
    ${EXTRA_ARGS} \
    "${GIT_ENVS[@]}" \
    $ENV_SUPPRESS \
    -w /workspace \
    --entrypoint /usr/local/bin/git-entrypoint.sh \
    -p "${HOST_PORT}:${CONTAINER_PORT}" \
    "$QWEN_IMAGE" \
    "$QWEN_CMD" $MODEL_CMD_ARGS
```

#### 10.7. Entry-point контейнера

**Файл:** `bin/docker-entrypoint.sh`

```bash
#!/bin/bash
export HOME=/workspace

# Symlink skills, если нужно
if [ ! -d "/root/.qwen/skills" ] && [ -d "/root/.qwen/shared-skills" ]; then
    ln -s shared-skills /root/.qwen/skills
fi

# Git config с хоста
[ -n "$GIT_CONFIG_NAME" ] && git config --global user.name "$GIT_CONFIG_NAME"
[ -n "$GIT_CONFIG_EMAIL" ] && git config --global user.email "$GIT_CONFIG_EMAIL"

exec "$@"
```

#### 10.8. Глобальная конфигурация — реализация

**Базовый путь:** `${XDG_CONFIG_HOME:-$HOME/.config}/${CONFIG_NAME}`

**Миграция:** функция `migrate_config()` — если существует `~/.qwen-docker` и не существует `~/.config/qwen-code-docker`, выполняется `mv`.

**Инициализация при `make install`:**

1. Создаются директории: `npm/`, `config/`, `skills/`
2. Скиллы из `config-templates/skills/*.md` копируются в глобальные `skills/` (с проверкой на существование)
3. `AGENTS.md` копируется в глобальные конфиги (с проверкой)
4. Создаётся symlink: `~/.local/bin/qwen` → `bin/qwen-run`

**Проектные конфиги:** создаются автоматически при запуске:

```bash
PROJECT_HASH=$(echo -n "$PROJECT_DIR" | md5sum | cut -d' ' -f1)
PROJECT_QWEN_DIR="$GLOBAL_CONFIG_DIR/projects/$PROJECT_HASH/.qwen"
mkdir -p "$PROJECT_QWEN_DIR"
```

#### 10.11. Безопасность — реализация

| Механизм                  | Реализация                                                                |
| ------------------------- | ------------------------------------------------------------------------- |
| Без привилегий            | `--security-opt no-new-privileges:true`                                   |
| Пользователь              | `--user $(id -u):$(id -g)` (Docker) / `--userns=keep-id` (Podman)         |
| SELinux                   | `--security-opt label=disable`                                            |
| Временные директории      | `--tmpfs /workspace/.npm:mode=755`, `--tmpfs /workspace/.config:mode=755` |
| `.qwen` в проекте         | Bind-mount в `PROJECT_QWEN_DIR` (не tmpfs, не создаётся в проекте)        |
| Read-only монтирование    | `skills:ro`, `AGENTS.md:ro`, `entrypoint:ro`                              |
| HOME переопределён        | `export HOME=/workspace` в entrypoint                                     |
| Подавление предупреждений | `-e QWEN_ALLOW_RUN_IN_HOME=1`                                             |

#### 10.12. Скиллы — реализация

**File Consistency Checker** (`config-templates/skills/file-consistency-checker.md`):

- Проверяет наличие ключевых файлов проекта
- Запускает `bash -n` для проверки синтаксиса скриптов
- Проверяет исполняемость `bin/qwen-run` и `bin/docker-entrypoint.sh`
- Сравнивает шаблоны `config-templates/` с ожидаемым состоянием
- Выводит итоговый отчёт с ✅/❌

**Git Commit** (`config-templates/skills/git-commit.md`):

- Триггер: ключевые слова («закоммить», «commit» и т.д.)
- Шаг 1: `git status && git diff HEAD && git log -n 3 --oneline`
- Шаг 2: определение типа по Conventional Commits
- Шаг 3: `git add` конкретных файлов (не `-A`)
- Шаг 4: генерация черновика сообщения (заголовок en, тело ru)
- Шаг 5: запрос подтверждения
- Шаг 6: `git commit`
- Шаг 7: проверка `git status && git log -n 1`
- Edge cases: нет изменений, нет staged, конфликты, нет git identity, секреты

#### 10.13. AGENTS.md — реализация

Файл содержит 7 разделов инструкций для AI:

1. **Роль** — AI-ассистент в изолированном контейнере
2. **Правила безопасности** — не запрашивать секреты, не читать за пределами `/workspace`, уважать `.qwenignore`
3. **Правила работы с кодом** — стиль проекта, комментарии, тесты
4. **Доступные инструменты** — только `/workspace`, стандартные утилиты
5. **Структура проекта** — исследовать через `ls`/`find`
6. **Стиль общения** — краткость, язык пользователя
7. **Git commit messages** — Conventional Commits, заголовок en, тело ru

### 12. Глоссарий

| Термин                   | Определение                                                                                       |
| ------------------------ | ------------------------------------------------------------------------------------------------- |
| **Перекрытие (overlay)** | Блокировка доступа AI к файлу через монтирование пустого Docker volume поверх существующего файла |

| **XDG Base Dir**         | Стандарт размещения пользовательских конфигов (`~/.config/`)                                      |
| **Runtime**              | Docker или Podman — среда выполнения контейнеров                                                  |
